# Feature 开发技能（拆分版）

## ⚠️ 重要指令

**不要并发执行步骤,后面步骤必须要前面步骤完全执行成功**

**DO NOT ENTER PLAN MODE** - 此技能要求直接执行，不进入计划模式。

**记忆查询**: 在需要回忆过往记录时（如查找相似功能、历史修复、相关笔记），可使用 `memory-query` 技能进行查询减少上下文。

**开发记录**: 阶段1后的每一步的关键结果记录到阶段1创建的[<feature_id>-开发记录]note中
    - 如果记录时因为入参不规范导致失败,需要重试

**开发流程中会创建的记录**
  feature:
    id:
      [feature_id]
  note:
    summary:
      [<feature_id>-需求确认] 
      [<feature_id>-实现方案]
      [<feature_id>-开发记录]
    
**进度恢复**: 如果用户指定了已经记录功能和有该功能的开发记录的情况下，根据开发记录直接恢复之前上下文，然后跳到记录阶段向后执行。
  - 如果不确定是哪一步则提示用户从阶段1开始,用户确认后清理开发记录后从阶段1开始

**开始前需要建立TODO列表让用户知道当前到那个阶段的那个步骤了**

## 流程概览

```
                                                                                                                                     
阶段1    →  阶段2    →  阶段3    →  阶段4    →  阶段5    →  阶段6    →  阶段7    →  阶段8    →  阶段9    →  阶段10   →  阶段11   →  阶段12   →  阶段13
项目确认 → 回忆规范 → 相关探索 → 需求澄清 → 需求确认 → 方案设计 → 方案选择 → 测试确认 → Git准备  → 代码修改 → 单元测试 → 全量测试 → 分支合并
                                                                                                |        |           |
                                                                                                ------<--失败---<-----
```

---

## 阶段 1: 项目确认

1. 按照 项目确认.md 执行

2. 建立feature
    - summary:<用户需求摘要>
    - content:<用户需求>
    - 如果用户需求超出content限制则content也记录摘要然后建立note[<feature-id>-初始需求]进行记录
    
3. 建立note命名[<feature_id>-开发记录]

---

## 阶段 2: 回忆开发所用规范

1. 使用技能`rememory-std` 项目结构规范 项目规范 代码开发相关规范

展示摘要：
```
[规范的summary] [规范的content（太长省略用...隐藏后续内容）]
[规范的summary] [规范的content（太长省略用...隐藏后续内容）]
...
```

---

## 阶段 3: 相关性探索

1. 相关功能探索
    1) 回忆相关项目功能
    2) 确认相关功能涉及代码范围
      - 如果没有回忆到则进行代码探索来完成相关功能的探索
    3) 探索这些范围代码
       - 确认需求新旧界限（新功能添加还是旧功能优化）
           - 新功能是新到什么地步，数据操作链路需要如何新增
               - 需要的基础脚手架有没有
               - 现有框架引用库的情况下能不能轻松实现
               - 需要不需要引用新的库
           - 旧功能优化（已有的数据操作链路完全支持）
               - 确认使用的共用函数和类
               - 确认使用的外部库
    4) 从而弄清楚当前需求基础背景

根据探索**展示**对需求的初步理解
```
需求:xxx

功能新旧:xxx

数据操作链路:xxx

文件修改范围:xxx

模糊点:
  1.
  2.
```

---

## 阶段 4: 需求澄清

1. 按照 ./需求澄清.md 执行

---

## 阶段 5: 需求确认

1. 按照 ./需求确认.md 执行

2. 更新之前记录的feature
    - content：为确认的需求摘要
    
3. 建立note[<feature_id>-需求确认] 记录完整需求

---

## 阶段 6: 方案设计

1. 按照 ./方案设计.md 执行

2. 请问用户有没有方案设计方向建议,
  - 如果提出建议则返回当前阶段的1步骤重新设计方案
  
3. 要求用户 **选择一个方案**

4. 进入下一步

---

## 阶段 7: 选择方案进行详细规划

1. 按照 ./方案规划.md 执行

2. 建立note[<feature_id>-实现方案] 记录完整方案内容

---

## 阶段 8: 测试确认

1. 使用技能`rememory-std` 测试规范
```
展示摘要：
[规范的summary] [规范的content（太长省略用...隐藏后续内容）]
[规范的summary] [规范的content（太长省略用...隐藏后续内容）]
...
```

2. 根据测试规范，确认的需求和选择的方案设计针对本次功能的单元测试文件。

3. 要求阐述清楚测试步骤，测试用例和边界情况。

4. 等待用户确认，如果用户要求修改则根据用户修改后回到本步骤第2项重新确认测试内容

---

## 阶段 9: Git 分支准备

1. 根据 ./Git分支准备.md 执行

---

## 阶段 10: 代码修改

**开始修改前建议用户 accept edits 模式**

1. 要求按照确认的方案进行代码编写,如果是测试返回的代码修改,则根据具体情况进行修改
  - 每完成一个功能模块后更新开发记录note,记录摘要
  - 如果编写阶段出现文件修改方面的问题,需要先解决该问题后再修改代码.
    - 多次解决不掉则让用户协助

2. 全部修改完成后,如果有则使用对应语言的静态检测工具进行扫描
    - 如果有ERROR类型的扫描结果,则修复然后返回本阶段2步骤
      - 循环三次后暂停请求用户协助


---

## 阶段 11: 本次需求的单元测试 

1. 根据之前确认的需求单元测试创建单元测试文件然后执行测试

2. 如果测试失败: 
    - 如果因为需求实现代码逻辑问题 **返回阶段10** 进行分析错误原因并修改代码
    - 如果因为单元测试文件本身问题修改后重新执行
    - 不管什么原因循环超过3次仍然失败，暂停并请用户介入决策

3. 单元测试成功

**本阶段结束需要退出 accept edits 模式**

---

## 阶段 12: 项目的全测试用例测试

1. 检查项目测试规范，如果有则执行项目全用例测试
    - 如果本次修改导致测试需要调整则需要 **返回阶段10** 进行测试代码调整

2. 如果测试失败 **返回阶段10** 进行分析错误原因并修改代码

3. 循环超过3次仍然失败，暂停并请用户介入决策

---

## 阶段 13: Git 分支合并

1. 按照 ./Git分支合并.md 执行

2. 合并完成后更新feature状态为completed

3. 整理开发记录note，补充合并信息
