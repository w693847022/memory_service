# Bug 修复技能

## ⚠️ 重要指令

**DO NOT ENTER PLAN MODE** - 此技能要求直接执行，不进入计划模式。

此技能定义了完整的 Bug 修复流程，包括问题分析、根因定位、代码修复和验证。按照以下阶段直接执行即可，无需使用 EnterPlanMode。

**记忆查询**: 在需要回忆过往记录时（如查找相似 Bug、历史修复、相关笔记），可使用 `memory-query` 技能进行查询。

---

结构化 Bug 修复流程，从问题分析到修复验证，通过 Memory MCP 记录全过程。

---

## 流程概览

```
回忆规范 → 项目确认 → 相关性探索 → 问题分析 → 问题确认 → 根因定位 → 方案设计 → 方案选择 → Git分支准备 → 代码修复 → 测试验证 → Git分支合并 → 完成记录
   │          │          │          │          │          │          │          │          │           │          │           │           │
   └─调用技能─→ 确定项目 ─→ Memory查询 ─→ 精准提问 ─→ Memory记录 ─→ 直接分析 ─→ 方案设计 ─→ 用户选择 ─→ 调用技能 ─→ Auto Accept ─→ 测试通过 ─→ 调用技能 ─→ 更新状态
```

## 阶段 0: 回忆项目规范

**目标**: 在开始 Bug 修复前，回忆相关项目规范

**流程**:
1. 调用 `rememory-std` 技能获取项目规范
2. 根据修复类型选择合适的规范参数：
   - 代码修复 → `Skill: rememory-std, args: "代码规范"` 或 `"开发规范"`
   - API 修复 → `Skill: rememory-std, args: "API规范"`
   - 配置修复 → `Skill: rememory-std, args: "开发规范"`
   - 不确定类型 → `Skill: rememory-std` (获取所有规范)
3. 解析返回的 JSON 格式规范，插入到上下文中
4. 确保后续修复符合项目规范

**规范检查要点**:
- 代码风格和命名规范
- API 设计规范
- 项目结构规范
- 测试规范

## 阶段 0.5: 项目确认

**目标**: 确认当前操作的项目，为后续相关性探索提供准确的项目名称

**流程**:
1. 调用 `project_list()` 获取所有已注册项目
2. 根据当前工作目录或用户输入，确定目标项目
3. 保存 `project_id` 和 `project_name`，供后续阶段使用

**确认方式**:
- **方式 1（自动匹配）**: 如果当前工作目录匹配某个项目的 path，自动选择
- **方式 2（用户选择）**: 如果有多个候选项目，展示列表供用户选择
- **方式 3（用户输入）**: 如果没有匹配项目，提示用户输入项目名称或创建新项目

**输出示例**:
```
项目确认：
- 当前项目: <项目名称> (ID: <project_id>)
- 项目路径: <项目路径>
```

**如果项目不存在**:
- 询问用户是否需要注册新项目
- 调用 `project_register()` 注册项目

**阶段自检**:
- [ ] project_list() 调用成功
- [ ] project_id 已确定
- [ ] project_name 已确定

## 阶段 1: 相关性探索

**目标**: 在提问前，查询过往记录，了解是否有相似 Bug/相关功能

**流程**:
1. **调用 memory-query 技能**（推荐）：
   - 使用阶段 0.5 确认的 `project_name` 作为第一个参数
   - 使用 Bug 相关关键词作为第二个参数
   - 示例：`Skill: memory-query, args: "<project_name> <搜索关键词>"`
2. 或者**直接使用 Memory MCP** 查询：
   - 使用阶段 0.5 确认的 `project_id`
   - 查询关键词：
     - Bug 相关的关键词（如"登录""超时""崩溃"等）
     - 模块相关的标签（如"auth""api""ui"等）
3. 分析查询结果，提取有价值信息：
   - 相似 Bug 的修复方案
   - 相关功能的实现细节
   - 历史问题或注意事项

**查询示例（使用 memory-query 技能）**:
```
# 阶段 0.5 已确认 project_id 和 project_name
# 调用 memory-query 技能，传入项目名称和搜索关键词
Skill: memory-query, args: "<project_name> <搜索关键词>"
```

**查询示例（直接使用 Memory MCP）**:
```
# 使用阶段 0.5 确认的 project_id
# 1. 查询 fixes 分组（查找相似 Bug）
project_get(
    project_id="<已确认的project_id>",
    group_name="fixes"
)

# 2. 查询 features 分组（查找相关功能）
project_get(
    project_id="<已确认的project_id>",
    group_name="features"
)

# 3. 查询 notes 分组（查找修复方案）
project_get(
    project_id="<已确认的project_id>",
    group_name="notes"
)

# 4. 分析结果，寻找与当前 Bug 相关的记录
```

**相似记录判断标准**:
- Bug 标题/关键词相似度 > 70%
- 问题涉及的功能模块/代码位置相同
- 涉及相同的错误类型或异常
- 示例：已有"登录接口超时问题"，新问题"登录接口响应慢" → 视为相似

**探索结果处理**:
- 如果找到相似 Bug：记录 `similar_fix_id`，在后续阶段中复用
- 如果找到相关功能：记录 `related_feature_id`，用于关联
- 如果找到相关笔记：记录 `similar_note_id`，参考历史修复方案
- 如果未找到相似记录：标记为新 Bug，后续创建新记录

**阶段自检**:

## 阶段 2: 问题分析

**目标**: 基于相关性探索结果，通过 1-3 轮精准提问，明确 Bug 的具体表现和影响范围

**流程**:
1. 分析用户提供的 Bug 描述
2. 结合相关性探索结果，识别：
   - 已明确的事项（跳过）
   - 仍需澄清的问题点
   - 复现步骤、错误信息、影响范围
3. 每轮提问 2-4 个**针对性**问题
4. 用户回答后，判断是否需要继续澄清

**提问原则**:
- **优先利用历史信息**: 如果相似 Bug 已有明确信息，不再重复询问
- **聚焦差异点**: 针对当前 Bug 与已有记录的差异进行提问
- **渐进式深入**: 先问核心问题，再问细节

**提问示例（有相似记录）**:
```
问题分析 (第 1/3 轮):

参考过往记录："登录接口超时问题" 已解决超时问题。

针对当前"登录接口响应慢"问题，需要澄清：
1. 本次响应慢的具体表现是什么？（与超时问题有何不同）
2. 响应时间大概是多少？
3. 是所有用户都受影响还是特定用户？
4. 问题发生的频率如何？
```

**提问示例（无相似记录）**:
```
问题分析 (第 1/3 轮):
1. Bug 的复现步骤是什么？
2. 预期行为是什么？实际行为是什么？
3. 是否有错误日志或异常信息？
4. 这个 Bug 影响哪些用户或功能模块？
```

**分析完成标准**:
- 复现步骤清晰
- 预期与实际行为对比明确
- 影响范围确定
- 严重程度评估完成

**阶段自检**:
- [ ] 2-4 个精准问题已提出
- [ ] 用户回答已收集
- [ ] 复现步骤已明确
- [ ] 预期行为和实际行为已对比
- [ ] 影响范围已确定
- [ ] 严重程度已评估（critical/high/medium/low）

**输出进度**:

#   - 如果发现相似 Bug：
#     - 询问用户是否复用该记录（展示已有记录的 content 和状态）
#     - 用户确认复用：使用已有 fix_id，跳过创建步骤

# 4. 如果需要，注册新标签
tag_register(
    project_id="<项目ID>",
    tag_name="<标签名>",
    description="<标签语义描述(10-50字)>"
)

# 5. 创建 Fix 记录（仅在无相似记录或用户不复用时）
# 准备 related_feature 参数（如果用户选择关联）
project_add(
    project_id="<项目ID>",
    group="fixes",
    content="<Bug标题>: <问题描述摘要>",
    description="<详细问题描述>",
    severity="<严重程度>",
    tags="<模块>,bug,<严重程度>",

)

| high | 核心功能不可用、严重性能问题 |
| medium | 功能异常但有替代方案、体验问题 |
| low | 轻微问题、UI瑕疵、非关键功能 |

### 6.3 记录 Note

将方案和测试用例记录为 Note：

```
# 【重要】首先检查 Fix 是否已有关联的 Note
# 获取当前 Fix 详情，查看 note_id 字段
current_fix = project_get(
    project_id="<项目ID>",
    group_name="fixes",
    item_id="<fix_id>"
)
existing_note_id = current_fix.get("note_id")

if existing_note_id:
    # Fix 已有关联 Note
    existing_note = project_get(
        project_id="<项目ID>",
        group_name="notes",
        item_id=existing_note_id
    )
    # 询问用户：
    # - 选项1：更新现有 Note (推荐)
    # - 选项2：创建新 Note (会替换关联)

    if 用户选择更新现有:
        project_update(
            project_id="<项目ID>",
            group="notes",
            item_id=existing_note_id,
            content="<追加新方案和测试用例>",
            description="<更新后的描述>",
            tags="<追加标签>"
        )
        note_id = existing_note_id
    else:
        # 创建新 Note
        project_add(...)
        note_id = 新note_id
else:
    # Fix 无关联 Note，检查是否有相似 Note
    project_get(
        project_id="<项目ID>",
        group_name="notes"
    )
    # 分析已有 notes 列表，检查是否有与当前 Bug 相关的修复方案记录
    # 如果找到相似记录，询问用户是否复用

    # 创建新 Note
    project_add(...)
    note_id = 新note_id

# 关联 Fix 和 Note（如果 note_id 变化了）
project_update(
    project_id="<项目ID>",
    group="fixes",
    item_id="<fix_id>",
    note_id=note_id
)
```

**Note 处理原则**:
1. **优先更新现有**: 如果 Fix 已有关联 Note，优先更新而不是新建
2. **保持历史**: 追加内容而不是替换，保留历史修复记录
3. **清晰分隔**: 使用 `---` 或日期标记分隔不同的修复版本

**相似 Note 判断标准**:
- description 中包含相同的 Bug 标题或关键词
- tags 中有相同的 bug/fix 相关标签
- 示例：已有"登录超时 - 修复方案"，新问题"登录响应慢" → 可更新同一 Note

## 标签处理流程

```
1. 调用 project_tags_info(project_id) 获取已注册标签
2. 根据 Bug 类型匹配合适的已注册标签
3. 若无合适标签:
   - 调用 tag_register() 注册新标签
   - 标签名称使用英文，描述10-50字
4. 使用选定的标签记录 Fix/Note
```

**常用标签示例**:
| 标签名 | 语义描述 |
|--------|----------|
| bug | Bug 修复相关 |
| fix | 代码修复 |
| critical | 严重级别问题 |
| high | 高优先级问题 |
| medium | 中优先级问题 |
| low | 低优先级问题 |
| api | API 相关问题 |
| ui | 界面相关问题 |
| core | 核心模块问题 |
| security | 安全相关问题 |
| performance | 性能相关问题 |

## 异常处理

| 场景 | 处理方式 |
|------|----------|
| 用户取消流程 | 询问是否保留已记录的 Fix |
| 无法复现 Bug | 记录问题状态，请求更多信息 |
| 根因定位困难 | 扩大搜索范围，请求用户提供更多信息 |
| 修复引入新问题 | 回滚修改，重新评估方案 |
| 测试失败 | 记录失败原因，继续修复或调整方案 |
| 需要修改数据库 | 评估数据迁移风险，提供迁移脚本 |

## 与 Feature 开发的区别

| 方面 | Bug 修复 | Feature 开发 |
|------|----------|--------------|
| 记录类型 | fixes | features |
| 优先评估 | 严重程度 | 功能优先级 |
| 分析重点 | 根因定位 | 需求澄清 |
| 方案侧重 | 最小化修改、避免回归 | 功能完整性 |
| 测试重点 | 回归测试、边界测试 | 功能测试、集成测试 |

## 注意事项

1. **禁止进入 Plan 模式**: 本技能已包含完整的修复流程，直接按照阶段执行即可。使用 Explore 子代理进行代码探索和根因定位。
2. **标签先检查再注册**: 避免重复注册
3. **保持记录关联**: Fix 和 Note 必须正确关联
4. **进度透明**: 每个阶段都显示进度状态
5. **回归测试优先**: 修复后必须验证现有功能不受影响
6. **最小化修改**: 只修改必要的代码，避免过度重构
7. **严重程度评估**: critical/high 级别的 Bug 需要优先处理
