# 代码实现与测试技能

## 参数规范

| 参数 | 说明 | 要求 |
|------|------|------|
| `item_id` | 已存在的 feature 或 fix ID | **必需** |
| `--plan` | 直接提供完整修改方案（可选） | 可选，不提供则从 memory_mcp 查询 |

**参数规则**:
- 无参数时：拒绝执行
- 仅提供 `item_id`：从 item_id 记录的 content 和关联 item 中获取完整修改方案
- 提供 `--plan`：使用用户提供的方案，跳过 memory_mcp 查询

---

## 前置条件

### 必需前置条件
- `item_id` 必须存在（features 或 fixes 分组）
- 必须存在以下之一：
  - **方式1（推荐）**: item_id 记录的 content 或关联 item 中包含完整修改方案
  - **方式2**: 用户通过 `--plan` 参数或交互式输入提供完整修改方案

### Git 分支要求
- **当前分支不能是 `main` 或 `master`**
- **分支必须是干净的**（无未提交的更改）
- 如果不满足，技能将拒绝执行并提示用户

### 完整修改方案格式要求
```markdown
# <item_id> 实现方案

## 需求/问题描述
[完整的需求或 bug 描述]

## 代码修改计划
### 修改的文件列表
- `path/to/file1.ts`: 修改说明
- `path/to/file2.ts`: 修改说明

### 具体修改内容
[详细的代码修改方案，包括函数签名、逻辑变更等]

## 测试计划
### 单元测试
- [ ] 测试场景1: 描述
- [ ] 测试场景2: 描述

### 整合测试
- [ ] 测试场景1: 描述
```

---

## ⚠️ 重要指令

**DO NOT ENTER PLAN MODE** - 此技能要求直接执行实现方案，不进入计划模式

**使用子代理处理 memory_mcp 和文件操作** - 以下操作通过 Agent 工具执行，减少主窗口上下文污染：
- 所有 `mcp__memory_mcp__*` 工具调用
- 所有文件操作：`Read`、`Edit`、`Write`、`Glob`、`Grep`

**循环计数** - 所有循环重试最多3次，超限后暂停并请求用户介入

**提醒用户使用 auto 模式** - 技能开始时提醒用户可使用 auto 模式提高效率

## 初始化阶段

### 步骤 1: 参数验证与方案获取

1. **验证 item_id 并获取记录**
   - 使用子代理查询 memory_mcp，获取 `item_id` 的完整记录（features 或 fixes 分组）
   - 如果不存在，拒绝执行并提示

2. **获取完整修改方案**（按优先级尝试）

   **方式1**: 从 item_id 记录的 content 中获取
   - 检查 item_id 的 content 字段是否包含完整的修改方案
   - 如果包含，验证方案格式是否完整（包含代码修改计划和测试计划）

   **方式2**: 从关联 item 中获取
   - 如果 content 中没有完整方案，检查 `related` 字段
   - 遍历关联的 item_id，查询其 content
   - 查找包含完整修改方案的关联 item
   - 如果找到，使用该方案

   **方式3**: 使用用户提供的方案
   - 如果以上方式均未找到，检查是否提供了 `--plan` 参数
   - 使用用户提供的方案，验证格式完整性

3. **方案获取失败处理**
   - 如果所有方式都无法获取完整修改方案，拒绝执行并提示：
     ```
     错误: 无法获取完整修改方案
     请确保以下之一:
       1. item_id 的 content 包含完整修改方案
       2. 关联 item 的 content 包含完整修改方案
       3. 使用 --plan 参数提供方案
     ```

### 步骤 2: Git 分支检查

1. **检查当前分支**
   ```bash
   git branch --show-current
   ```
   - 如果是 `main` 或 `master`，拒绝执行并提示：
     ```
     错误: 当前在 main/master 分支上
     请先创建并切换到开发分支，例如：
       git checkout -b feature/<item_id>
       git checkout -b fix/<item_id>
     ```

2. **检查分支状态**
   ```bash
   git status --porcelain
   ```
   - 如果有未提交的更改，拒绝执行并提示：
     ```
     错误: 当前分支有未提交的更改
     请先提交或暂存更改：
       git stash    # 暂存更改
       git commit   # 或提交更改
     ```

### 步骤 3: 提醒用户

```
提示: 建议使用 auto 模式以提高效率
当前已进入实现阶段，将自动执行代码修改和测试
```

## 阶段 1: 代码实现

**目标**: 按照确认的方案进行代码编写

**流程**:

### 步骤 1.1: 读取并解析修改方案

从获取的完整修改方案中提取：
- 需修改的文件列表
- 每个文件的具体修改内容

### 步骤 1.2: 执行代码修改

对每个需要修改的文件：

1. **使用 Read 工具读取文件**
2. **按照方案进行修改**：
   - 使用 Edit 工具进行精确修改
   - 或使用 Write 工具进行重写（如方案要求）
3. **验证修改**：读取修改后的文件确认更改正确

### 步骤 1.3: 静态代码检查（如适用）

根据项目语言执行静态检查：

| 语言 | 工具 | 命令 |
|------|------|------|
| Python | ruff | `ruff check .` |
| Python | mypy | `mypy .` |
| TypeScript | eslint | `eslint .` |
| Go | go vet | `go vet ./...` |

**处理逻辑**：
1. 执行静态检查
2. 如果有 **ERROR** 级别问题：
   - 修复问题 → 返回步骤 1.2
   - 循环计数 +1
   - 超过3次 → 暂停并请求用户协助
3. **WARNING** 级别问题：显示但不阻塞流程

---

## 阶段 2: 单元测试

**目标**: 根据测试计划创建并执行单元测试

**流程**:

### 步骤 2.1: 分析测试需求

从修改方案中提取：
- 需要新增/修改的测试文件
- 需要覆盖的测试场景

### 步骤 2.2: 创建/修改单元测试

1. **定位测试文件**：
   - 通常在 `tests/`、`__tests__/` 或与源码同目录的 `test_*.py`、`*.test.ts`
2. **编写测试代码**：
   - 按照测试计划覆盖所有场景
   - 确保测试可独立运行

### 步骤 2.3: 执行单元测试

**根据项目类型执行测试**：

| 项目类型 | 测试命令 |
|----------|----------|
| Python pytest | `pytest tests/ -v` |
| Python unittest | `python -m unittest discover` |
| Node.js | `npm test` |
| Go | `go test ./...` |

### 步骤 2.4: 处理测试失败

**失败分析**：
1. 判断失败原因：
   - **实现代码问题** → 返回阶段1步骤1.2，修复代码后重新测试
   - **测试代码问题** → 修复测试代码，重新执行
2. 循环计数 +1
3. 超过3次 → 暂停并请求用户介入决策

---

## 阶段 3: 整合测试

**目标**: 执行项目全量测试，确保修改未破坏现有功能

**流程**:

### 步骤 3.1: 检查项目测试规范

1. **检查是否存在测试脚本**：
   ```bash
   # 按优先级检查
   ls scripts/test.sh
   ls scripts/run_tests.sh
   cat package.json | jq .scripts.test
   ```
2. **如果存在测试规范**：执行项目全量测试
3. **如果不存在**：执行阶段2的单元测试命令作为替代

### 步骤 3.2: 执行整合测试

```bash
# 示例命令（根据项目实际情况）
./scripts/run_tests.sh
# 或
pytest tests/ --cov
# 或
npm test -- --coverage
```

### 步骤 3.3: 处理整合测试失败

**失败分析**：
1. 判断失败原因：
   - **本次修改导致现有测试失败** → 返回阶段1，分析并修改代码
   - **测试需要调整**（如API变更）→ 返回阶段1调整代码后更新测试
2. 循环计数 +1
3. 超过3次 → 暂停并请求用户介入决策

## 错误处理与恢复

### 循环超限处理

当任何阶段的循环次数超过3次时：

1. **暂停执行**
2. **向用户报告当前状态**：
   ```
   ⚠️ 循环次数已达上限 (3次)

   当前阶段: <阶段名称>
   当前问题: <问题描述>

   请选择:
   1. 继续重试
   2. 手动介入后继续
   3. 中止执行
   ```
3. **等待用户决策**

### 子代理使用模式

**memory_mcp 操作**：

```python
# 示例：查询 item 信息
Agent(
    description="查询 item 信息",
    prompt=f"""
    使用 mcp__memory_mcp__project_get 查询以下信息：
    - project_id: "ai_memory_mcp"
    - group_name: "features"
    - item_id: "{item_id}"

    返回完整结果。
    """,
    subagent_type="general-purpose"
)
```

**文件操作**：

```python
# 示例：读取并修改文件
Agent(
    description="读取并修改文件",
    prompt=f"""
    1. 使用 Read 工具读取 path/to/file.ts
    2. 使用 Edit 工具按照以下要求修改：
       - 将函数名从 oldFunc 改为 newFunc
       - 更新调用处
    3. 使用 Read 工具验证修改结果
    """,
    subagent_type="general-purpose"
)

# 示例：搜索文件
Agent(
    description="搜索相关文件",
    prompt="""
    使用 Glob 工具查找所有 .ts 测试文件：
    - pattern: "**/*.test.ts"

    返回找到的文件列表。
    """,
    subagent_type="general-purpose"
)
```
