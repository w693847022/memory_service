# Feature 开发技能

## ⚠️ 重要指令

**DO NOT ENTER PLAN MODE** - 此技能要求直接执行，不进入计划模式。

此技能定义了完整的功能开发流程，包括需求澄清、方案设计、代码实现和记录。按照以下阶段直接执行即可，无需使用 EnterPlanMode。

**每步检测机制**: 每个阶段完成后必须通过自动检测才能进入下一步，检测失败时停在当前步等待修复。

所有的信息/代码搜索进入子代理模式进行

**记忆查询**: 在需要回忆过往记录时（如查找相似功能、历史修复、相关笔记），可使用 `memory-query` 技能进行查询减少上下文。

---

## 流程概览

```
回忆规范 → 项目确认 → 相关性探索 → 需求澄清 → 需求确认 → 方案设计 → 方案选择 → 测试确认 → 方案记录 → Git分支准备 → 代码实现 → Git分支合并 → 完成记录
   │          │          │          │          │          │          │          │          │          │           │          │           │
   └─调用技能─→ 确定项目 ─→ Memory查询 ─→ 精准提问 ─→ Memory记录 ─→ 直接设计 ─→ 用户选择 ─→ 用户确认 ─→ Memory记录 ─→ 调用技能 ─→ Auto Accept ─→ 调用技能 ─→ 更新状态
```

## 阶段 0: 回忆项目规范

**目标**: 在开始功能开发前，回忆相关项目规范

**流程**:
1. 调用 `rememory-std` 技能获取项目规范,回忆尽量全面,可多次反复。
2. 根据功能类型选择合适的规范参数：
   - 代码功能 → `Skill: rememory-std, args: "代码规范"` 或 `"开发规范"`
   - API 功能 → `Skill: rememory-std, args: "API规范"`
   - 文档相关 → `Skill: rememory-std, args: "文档规范"`
   - 不确定类型 → `Skill: rememory-std` (获取所有规范)
3. 解析返回的 JSON 格式规范，插入到上下文中
4. 确保后续开发符合项目规范

**规范检查要点**:
- 代码风格和命名规范
- API 设计规范
- 项目结构规范
- 测试规范

**规范展示**:
```
调用 rememory-std 后，展示规范摘要：

## 项目规范摘要

**代码规范**:
- <代码风格要点>

**API规范**:
- <API设计要点>

**项目结构**:
- <项目结构要点>

**测试规范**:
- <测试要点>
```

**无规范提醒**:
如果未获取到规范，显示：
```
⚠️ 未检查到项目规范
提示: 项目可能尚未设置规范，或规范内容为空。
      继续开发时如遇规范相关问题，请联系项目管理员设置规范。
```

**输出进度**:
```
实现进度：
[✓] 回忆项目规范
[→] 项目确认
```

## 阶段 0.5: 项目确认

**目标**: 确认当前操作的项目，为后续相关性探索提供准确的项目名称

**流程**:
1. 调用 `project_list()` 获取所有已注册项目
2. 根据当前工作目录或用户输入，确定目标项目
3. 保存 `project_id` 和 `project_name`，供后续阶段使用

**确认方式**:
- **方式 1（自动匹配）**: 如果当前工作目录匹配某个项目的 path，自动选择
- **方式 2（用户选择）**: 如果有多个候选项目，展示列表供用户选择
- **方式 3（用户输入）**: 如果没有匹配项目，提示用户输入项目名称或创建新项目

**如果项目不存在**:
- 询问用户是否需要注册新项目
- 调用 `project_register()` 注册项目

**🔍 自动检测**:
```
检测内容: project_id 和 project_name 已确定
检测方式: 调用 project_get(project_id) 验证项目存在
检测通过标准: project_get 返回有效项目信息，包含 name 和 path 字段
```

**检测失败处理**: ⏸️ **检测失败，暂停等待用户处理**
- 失败原因: 项目不存在或 project_get 返回无效信息
- 处理方式: 请用户确认项目是否存在，或注册新项目

**输出示例**:
```
项目确认：
- 当前项目: <项目名称> (ID: <project_id>)
- 项目路径: <项目路径>

✅ 阶段0.5检测通过

实现进度：
[✓] 回忆项目规范
[✓] 项目确认 [检测通过]
[→] 相关性探索
```

## 阶段 1: 相关性探索

**目标**: 在提问前，查询过往记录，了解是否有相似功能/笔记

**流程**:
1. **调用 memory-query 技能**（推荐）：
   - 使用阶段 0.5 确认的 `project_name` 作为第一个参数
   - 使用功能相关关键词作为第二个参数
   - 示例：`Skill: memory-query, args: "<project_name> <搜索关键词>"`
2. 分析查询结果，提取有价值信息：
   - 相似功能的实现方案
   - 相关的架构决策
   - 历史问题或注意事项

**相似记录判断标准**:
- 功能名称/关键词相似度 > 70%
- 功能描述的核心概念一致
- 涉及相同的模块或技术栈
- 示例：已有"用户登录功能"，新需求"用户登录优化" → 视为相似

**探索结果处理**:
- 如果找到相似记录：记录 `similar_feature_id` 和 `similar_note_id`，在后续阶段中复用
- 如果未找到相似记录：标记为全新功能，后续创建新记录

**🔍 自动检测**:
```
检测内容: memory-query 已调用，探索结果已分析记录
检测方式: 检查是否获得了探索结果（相似记录或"未发现相似"的明确标记）
检测通过标准: 返回结果中包含相似功能列表、相关笔记列表或明确的"无相似记录"标记
```

**检测失败处理**: ⏸️ **检测失败，暂停等待用户处理**
- 失败原因: memory-query 未返回有效探索结果
- 处理方式: 请用户确认搜索关键词是否合适，或重新调用 memory-query

**输出示例**:
```
相关性探索结果：
- 发现相似功能: <功能名称> (ID: <feature_id>)
  - 状态: <completed/in_progress>
  - 描述: <功能描述>
- 发现相关笔记: <笔记标题> (ID: <note_id>)
  - 内容概要: <实现要点>

✅ 阶段1检测通过

实现进度：
[✓] 回忆项目规范
[✓] 项目确认
[✓] 相关性探索 [检测通过]
[→] 需求澄清
```

## 阶段 2: 需求澄清

**目标**: 基于相关性探索结果，通过 1-3 轮精准提问，明确功能需求细节

**流程**:
1. 分析用户提供的功能描述
2. 结合相关性探索结果，识别：
   - 已明确的事项（跳过）
   - 仍需澄清的模糊点
   - 边界条件、依赖关系
3. 每轮提问 2-4 个**针对性**问题
4. 用户回答后，判断是否需要继续澄清

**提问原则**:
- **优先利用历史信息**: 如果相似记录中已有明确信息，不再重复询问
- **聚焦差异点**: 针对当前需求与已有记录的差异进行提问
- **渐进式深入**: 先问核心问题，再问细节

**澄清完成标准**:
- 功能边界清晰
- 输入输出明确
- 技术约束确认

**🔍 自动检测**:
```
检测内容: 用户已回答所有澄清问题，需求边界清晰
检测方式: 检查澄清完成标准是否全部满足
检测通过标准:
  ✓ 功能边界已明确描述
  ✓ 输入/输出格式已确认
  ✓ 技术约束已确认
  ✓ 用户确认需求已清晰
```

**检测失败处理**: ⏸️ **检测失败，暂停等待用户处理**
- 失败原因: 需求澄清未完成，功能边界、输入输出或技术约束仍有缺失
- 处理方式: 请用户继续回答澄清问题，直到需求清晰

## 阶段 6: Git 分支准备

**目标**: 在代码实现前准备 Git 开发分支

**流程**:
1. 此时已有 `feature_id`（新建的或已存在的）
2. 调用 `git-branch` 技能准备分支：
   ```
   Skill: git-branch, args: "prepare <feature_id> feature"
   ```
3. 技能会：
   - 检查当前分支是否为 main/master（如果是则拒绝）
   - 保存当前分支名
   - 创建或切换到开发分支 `feature-{id}`

**🔍 自动检测**:
```
检测内容: 开发分支已创建或切换成功
检测方式: 执行 git branch 或 git rev-parse --abbrev-ref HEAD 检查当前分支
检测通过标准:
  ✓ 当前分支名为 feature-{id} 格式
  ✓ 或开发分支已存在于本地
```

## 阶段 8: Git 分支合并

**目标**: 在代码实现完成后，合并开发分支回原分支

**注意**: 目标分支一般不能是 main/master 分支，如果是需要向用户确认。

**流程**:
1. 调用 `git-branch` 技能合并分支：
   ```
   Skill: git-branch, args: "merge <feature_id> feature"
   ```
2. 技能会：
   - 切换回起始分支
   - 拉取远程更新
   - 合并开发分支（尝试自动解决冲突）
   - 调用 git-commit 技能生成提交信息
   - 打 tag（使用开发分支名）
   - 删除开发分支

**🔍 自动检测**:
```
检测内容: 分支已成功合并到目标分支
检测方式: 执行 git log 或 git branch -a 检查合并状态
检测通过标准:
  ✓ 当前分支已切换回原分支（非 feature 分支）
  ✓ 原分支包含开发分支的提交
  ✓ 或 git merge --no-ff 确认合并成功
```

**检测失败处理**: ⏸️ **检测失败，暂停等待用户处理**
- 失败原因: Git 分支合并失败，可能存在合并冲突
- 处理方式: 请用户解决合并冲突或确认合并状态，然后继续

## 阶段 9: 完成记录

**目标**: 更新 Feature 状态为完成，整合实际修改内容到 Note

### 9.1 更新 Feature 关联的 Note

将实际实现内容整合记录到 Note 中：

```
# 1. 获取当前 Feature 关联的 Note
current_feature = project_get(
    project_id="<项目ID>",
    group_name="features",
    item_id="<feature_id>"
)
note_id = current_feature.get("note_id")

# 2. 获取原 Note 内容
if note_id:
    existing_note = project_get(
        project_id="<项目ID>",
        group_name="notes",
        item_id=note_id
    )
    original_content = existing_note.get("content", "")
else:
    original_content = ""

# 3. 整合实际修改内容到 Note
implementation_record = f"""
## 实现完成记录

**完成时间**: <YYYY-MM-DD HH:mm>

### 实际修改内容

<实际修改的文件列表>

### 实现差异说明

<方案设计与实际实现的差异对比>

### 合并信息

- 合并分支: feature-<id>
- 目标分支: <原分支名>
- Commit: <commit hash>

---

### 原方案记录
{original_content}
"""

# 4. 更新 Note（追加实现记录）
if note_id:
    project_update(
        project_id="<项目ID>",
        group="notes",
        item_id=note_id,
        content=implementation_record,
        tags="completed,<功能模块>"
    )
else:
    # 创建新的完成记录 Note
    project_add(
        project_id="<项目ID>",
        group="notes",
        summary="<功能名称> - 实现完成记录",
        content=implementation_record,
        tags="completed,note"
    )
    note_id = 新创建的note_id
    # 关联到 Feature
    project_update(
        project_id="<项目ID>",
        group="features",
        item_id="<feature_id>",
        note_id=note_id
    )
```

| 阶段 | 检测内容 | 检测方式 | 通过标准 |
|------|----------|----------|----------|
| 阶段0 | 无检测（展示规范摘要，无标准则提醒后继续） | - | - |
| 阶段0.5 | 项目ID和名称已确定 | project_get 验证 | 返回有效项目信息 |
| 阶段1 | 探索结果已分析 | 检查结果完整性 | 返回相似记录或"无相似"标记 |
| 阶段2 | 需求已澄清 | 检查澄清完成标准 | 功能边界、输入输出、技术约束已确认 |
| 阶段3 | Feature已创建 | project_get 验证 | feature_id存在，status="in_progress" |
| 阶段4 | 方案已设计 | 检查方案数量和字段 | 1-3个方案，每个包含必要字段 |
| 阶段5a | 用户已确认方案和测试用例 | 检查用户明确回答 | 用户已选择方案、确认测试用例、对详细实现无修改意见 |
| 阶段5b | Note已创建并关联到Feature | project_get 验证 | note_id存在且已关联到feature |
| 阶段6 | 开发分支已创建 | git branch检查 | 当前分支为feature-{id}格式 |
| 阶段7 | 代码实现且测试通过 | 文件存在+测试执行 | 核心文件存在，测试通过 |
| 阶段8 | 分支已合并 | git log/branch检查 | 原分支包含开发分支提交 |

## 异常处理

| 场景 | 处理方式 |
|------|----------|
| 检测失败 | 停在当前阶段，说明失败原因和修复方法，等待修复后重试 |
| 用户取消流程 | 询问是否保留已记录的 Feature |
| 方案设计阻塞 | 提供简化方案或请求用户指导 |
| 实现过程中发现问题 | 更新 Note 记录，保持 Feature 状态为 in_progress |
| 测试失败 | 停在阶段7，反馈失败原因，等待用户修复后继续 |

## 注意事项

1. **禁止进入 Plan 模式**: 本技能已包含完整的开发流程，直接按照阶段执行即可。进入 Plan 模式会清除上下文，导致 Memory MCP 记录中断。
2. **阶段0无检测**: 阶段0仅展示规范摘要，无标准时提醒用户但不阻塞流程。
3. **每步检测**: 除阶段0外，每个阶段完成后必须通过自动检测才能进入下一步，检测失败时停在当前步等待用户处理。
4. **标签先检查再注册**: 避免重复注册
5. **保持记录关联**: Feature 和 Note 必须正确关联
6. **进度透明**: 每个阶段都显示进度状态和检测结果
